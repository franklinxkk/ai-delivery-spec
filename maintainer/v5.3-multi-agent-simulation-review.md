## 1. 评审目标与方法

### 1.1 目标

本次评审不是验证 v5.3 设计“是否正确”，而是把自己放在一个真实的 Kimi k3 运行时中，用多个项目、多个阶段、多个角色的实际材料去“预演”v5.3 会怎么落地，找出设计在落地为代码、PRD、交接包和门禁规则时会遇到的**结构性短板**。

### 1.2 方法

- **输入**：v5.3 整合详细设计稿 + 本地 v5.x Skill 基线（`ai-delivery-spec-full/SKILL.md` 等） + 5 个真实项目材料。
- **模拟矩阵**：5 个场景 × 12 个角色/阶段切入点 = 12 个独立子代理并行推演。
- **判定标准**：每个子代理必须给出“按 v5.3 执行的推演路径 + 至少 3 条有证据的短板 + 角色评分 + 最高优先级修正建议”。
- **去偏措施**：子代理只读取指定场景文件，不互相共享结论；短板必须有 v5.3 设计原文或场景材料支撑。

### 1.3 场景选择理由

| 场景 | 代表类型 | 关键挑战 |
|---|---|---|
| 神通 CRM 经营中台 | L2 / Workflow / 15 模块 / 无 AI | 模块多但轻量、混合页面、移动外勤、跨模块共享规则 |
| 运智管家 AI 智能体 | L3 / AI-Native / GovTech / 交通运输安全 | AI Runtime Contract、法规溯源、数据隔离、安全生产 |
| 安驾融合出版与服务平台 | L2 兼 governed_truth 候选 / 内容版权 / 多渠道 | 多来源权威、复合视图、跨模块事件、工程基线 |
| TASK-01 RBAC + 账号管理 | 小范围横切关注点 / 已有代码原型 | 权限/租户/审计横切、代码先行、快速入项 |
| 运管数据集市 v7/v8 | Stage 0 逆向工程 / 数据密集型 / AI 配表 | 数据模型血缘、模板构建器、多版本原型 |

## 2. 总体结论

### 2.1 设计优势（保持不变）

v5.3 在以下方向上是正确且必要的：

1. **静默双轴路由** 显著降低新手认知负担；
2. **统一 PRD + 页面 profile** 能减少机械 N/A 和模板膨胀；
3. **单一确定性 gate 内核 + 四状态** 是防止 Agent 自证合格的关键；
4. **权威模型三元拆分**（写入面 / 约束来源 / 基线审批）比旧 `authority_mode` 更精确；
5. **运行包与完整仓库双通道** 解决了文件数膨胀和平台限制问题；
6. **中文 finding** 对中文产品团队更友好。

### 2.2 关键风险（必须修正）

但在 12 个模拟中，所有角色评分集中在 **6/10 左右**，没有一个场景达到 7.5 分以上。共性扣分点高度集中：

- **页面 profile  taxonomy 无法覆盖真实项目中的复合页面**（CRM 驾驶舱、安驾审核中心、数据集市模板构建器）；
- **模块 work packet 假设垂直业务切片，无法表达 RBAC、AI Runtime、合规审计等横切关注点**；
- **gate 只检查 PRD/页面结构，不检查 testability、跨模块边、数据模型、AI Runtime Contract、代码级安全**；
- **Coding Agent 交接缺少工程基线（TASK-00）入口和模块 handoff gate**；
- **权威模型对已有 PRD/代码先行场景 retrofit 成本过高**；
- **P0 unknown 二元判定与真实项目的分阶段未知项处理冲突**。

### 2.3 设计成熟度评分

| 维度 | 评分 | 说明 |
|---|---:|---|
| 入口简化（SKILL.md / 静默路由） | 7/10 | 方向对，但 130 行指标可能过激进 |
| 统一 PRD / 页面合同 | 6/10 | 结构好，profile 覆盖不足 |
| 权威模型 | 6/10 | 概念精确，落地摩擦高 |
| 大输入分轮 / 恢复 | 6/10 | 协议合理，数据密集型场景粒度冲突 |
| 确定性门禁 | 5/10 | 四状态正确，但检查面太窄 |
| Coding Agent 交接 | 5/10 | 有概念，缺可执行 schema 和 gate |
| 学习闭环 | 6/10 | 设计完整，但 v5.3 不应全部承载 |
| 发布工程 | 7/10 | allowlist 和双通道设计成熟 |
| **综合** | **6/10** | **能发布，但落地会频繁返工** |

## 3. 主题聚类：8 大系统性短板

### 3.1 页面 profile 无法覆盖复合视图与数据配置型 UI

**发现来源**：CRM:PM、CRM:CA、CRM:QA、Anjia:PM、Anjia:CA、Datamart:PM、Datamart:Architect

**问题**：v5.3 §7.2 定义 7 个 profile（dashboard/list/form/detail/workflow/composer/mobile），默认一个 `VIEW-*` 只声明一种。但真实项目中大量页面是混合的：

- CRM 老板驾驶舱 = dashboard + list + workflow + action_cards；
- 安驾审核中心 = queue/list + version_diff/detail + decision/workflow；
- 数据集市模板构建器 = 指标池拖拽 + 列映射 + 公式预览 + AI 推荐 + 填报/自动列混合，不是通用 composer。

**后果**：PM 被迫把本可一个视图讲清的页面拆成多个视图，或遗漏关键表面；Coding Agent 按单一 profile 实现后业务闭环缺失。

**建议**：
- 允许 `profile` 为数组或 `primary_profile + secondary_profiles`；
- 新增 `composite/portal` profile；
- 新增 `builder/designer` 或扩展 `composer-data` 数据语义表面；
- 扩展 `mobile` profile 的 `device_capabilities`（GPS、相机、录音、离线同步）。

---

### 3.2 模块 work packet 假设垂直切片，横切关注点无处安放

**发现来源**：CRM:CA、Anjia:Architect、Anjia:CA、RBAC:PM、RBAC:CA、AI-Agent:Architect

**问题**：v5.3 §8.5 推荐目录是 `modules/resource/AGENTS.md`、`modules/course/AGENTS.md` 等垂直模块。但以下能力横跨多个模块：

- RBAC/权限/数据范围/审计（安驾 TASK-01 的 `perm()` 守卫遍布所有路由）；
- AI Runtime Contract（运智管家 M17 智能体横跨 M01/M03/M09/M10/M11/M12/M15）；
- 数据隔离与租户隔离（运智管家 16 个模块共享）。

**后果**：若严格按模块切片，同一套约束被拆散，各 Agent 重复实现或漏掉；若塞进一个模块，又超出 15000 字符预算且边界模糊。

**建议**：
- 增加“横切关注点 packet”（`cross-cutting-security.md`、`cross-cutting-ai-runtime.md`）作为各模块 work packet 的必读附件；
- 对水平 AI 能力增加 `MOD-AI-xxx` 模块模式，只包含运行时契约、共享约束和 edge refs；
- work packet schema 增加 `ENTITY/REL/DATA-LINEAGE/DICT` 四类 ID，支持数据模型阶段。

---

### 3.3 确定性门禁检查面过窄

**发现来源**：CRM:QA、Anjia:Architect、Anjia:CA、RBAC:PM、RBAC:CA、AI-Agent:Compliance、Datamart:Architect

**问题**：v5.3 §10 的 gate 主要检查 PRD 结构、页面 profile、P0 unknown、引用一致性。但以下关键维度缺失：

- **Testability**：AC 覆盖率、状态 from/to/guard/action 矩阵、跨模块 E2E Canvas；
- **跨模块边**：producer/consumer、payload_version、idempotency_key_scope、compensation；
- **数据模型**：孤儿指标、悬空模板列、未解析报告变量；
- **AI Runtime**：confidence/fallback/human_gate/eval/observability；
- **实现层安全**：SQL 注入、参数化查询、输入校验；
- **隐性未知项**：“待实现”“TBD”“待确认”等未声明未知项。

**后果**：gate PASS 后，QA 仍无法写测试、架构师仍要人工核对跨模块边、Coding Agent 仍可能实现带 SQL 注入的代码。

**建议**：
- 对 L2+ `unified_prd` 增加 `testability` gate 分支；
- 对 `AI: true` 且 `assurance_profile=safety_critical/high_risk` 增加 AI Runtime Contract 强制检查；
- 增加 `cross-module-edge` 检查器和 `data-model` 检查分支；
- 增加“未知项扫描”：对 `TBD/待确认/待实现/stub` 等标记做确定性扫描，强制加入 `open_p0_unknown_ids`。

---

### 3.4 Coding Agent 交接缺少工程基线与 handoff gate

**发现来源**：Anjia:CA、RBAC:CA、CRM:CA

**问题**：v5.3 §8.5 假设“需求定稿后”直接生成模块 work packet。但真实项目中，Coding Agent 第一步必须先盘点现有仓库（安驾 handoff 明确要求 TASK-00）。v5.3 没有：

- 仓库基线发现阶段；
- 模块 work packet 的 schema 与版本锚定；
- 历史 GAP/P0 缺口到 REQ/AC/STATE 的映射检查；
- 项目级 coding 约束（禁止发明项）的校验。

**后果**：CA 拿到的工作包可能是草稿或旧版本，按错误切片实现后才发现与 PRD/验收不一致；或在错误技术假设上施工。

**建议**：
- 新增 `references/engineering-baseline.md`（或在 `tool-adapters.md` 中增加独立章节）；
- 定义 `module-work-packet.schema.json` 与 `project-handoff-gate`；
- 模块 `AGENTS.md` 模板前置 TASK-00 检查项：仓库基线、现有 auth/tenant/audit 模式、CFL/CHG 升级路径；
- work packet 强制携带 `project-coding-constraints.yaml`。

---

### 3.5 权威模型对已有项目 retrofit 成本过高

**发现来源**：CRM:PM、Anjia:PM、Datamart:Architect

**问题**：v5.3 §6.1 默认要求 `binding_sources` 每条来源都有 `source_ref`、`applies_to`、`precedence`、`interpretation_owner`，并要求 `DEC-AUTHORITY-001`。但真实项目中：

- CRM PRD 有 11 条 Kickoff 确认原则，无 SRC/DEC ID；
- 安驾 PRD §0.1 使用全局来源优先级（1–6 级），与 v5.3“禁止全局排行榜”冲突；
- 数据集市 39 个指标口径来源于线下 Excel 表，不是合同或客户确认。

**后果**：PM 需要把既有确认 retroactive 编码成治理文书，否则 gate BLOCK；或放弃权威模型，导致“变更如何同步”无人负责。

**建议**：
- 提供 `authority_bootstrap` 模式：允许已有 PRD 确认项以 `SRC-KICKOFF-00N` / `DEC-CONFIRM-00N` 批量登记，gate 只检查“每条确认项有来源/决策引用”；
- 增加 `imported_caliber_table` 类型 binding source，必填版本/哈希/导入周期/解释责任人；
- 对已有成熟统一 PRD 增加保留例外：若 PRD 已具备稳定 ID、机器投影、来源表，可由 DEC 明确保留 `canonical_authoring_surface=unified_prd`；
- 对法规类来源增加 `jurisdiction`、`regulatory_version`、`effective_date`、`sunset_date`。

### 3.6 P0 unknown 二元判定与分阶段未知项处理冲突

**发现来源**：Anjia:PM、Anjia:CA

**问题**：v5.3 §10.3 只要发现 `UNK-* + P0 + blocked/阻塞中` 就返回 `BLOCKED_BY_P0_UNKNOWN`。但安驾 PRD 明确区分：

- UNK-REPO-001：阻断具体实现，不阻断产品拆任务；
- UNK-ERP-001：不阻断；
- UNK-REGION-001：阻断该地区确认学时上线。

**后果**：gate 会误把已声明且已分阶段处理的未知项全部判为 BLOCKED，输出 false blocker，削弱门禁公信力。

**建议**：在未知项 schema 中增加 `phase_scope` / `blocks_stage[]` 字段；gate 仅当未知项影响当前阶段时才返回 `BLOCKED_BY_P0_UNKNOWN`。

---

### 3.7 路由 heuristics 存在误分类

**发现来源**：CRM:PM、Datamart:Architect、RBAC:PM、RBAC:CA

**问题**：

- v5.3 §5.2 “12 个以上模块”触发 `governed_truth`，会把模块多但轻量的 CRM 误判；
- v5.3 §8.1 “稳定对象 ≥ 200”不会触发大输入，但数据集市真正风险是指标×数据源表×维度血缘，不是对象数；
- RBAC 作为横切关注点，即使“Quick”也不应降级为 `requirement_card`。

**后果**：项目被推向过重或过轻的形态，增加路由摩擦或遗漏风险。

**建议**：
- 路由条件增加“模块耦合度/变更频率/审计要求/数据血缘”权重；
- 增加数据模型维度触发器：指标×数据源表×维度三元组超过阈值、跨模板/报告复用同一指标时建议 `governed_truth`；
- 增加横切关注点触发器：role/permission/tenant/audit/data_scope 跨 ≥3 个模块时，强制 `unified_prd` 并提示 `high_risk`。

---

### 3.8 AI Runtime 与 AI-core 模块缺少可施工 schema

**发现来源**：AI-Agent:Architect、AI-Agent:Compliance

**问题**：v5.3 §6.4 只列出 Profile A 触发条件，没有 Prompt/Tool/Eval/Rollback 的统一 schema；`binding_sources` 未覆盖 Prompt/Tool Registry；finding code 没有 AI 专用类别。

**后果**：Architect 需要为每个 AI 模块自行设计运行时契约，格式不统一，容易遗漏 confidence/fallback/human_gate/eval/observability 中某一项。

**建议**：
- 新增 `references/ai-runtime-contract.md` 触发式参考，提供统一 YAML schema；
- 把 Prompt/Tool Registry 纳入 `binding_sources`；
- 扩展 finding code 命名空间：`AI-CONTRACT-*`、`AI-PROMPT-*`、`AI-EVAL-*`、`AI-OBS-*`、`AI-FALLBACK-*`。

## 4. Greenfield 万亿市场领域补充模拟

### 4.1 模拟目的

除了工作区已有项目，还选取了 4 个万亿级市场、且不在现有 `references/domain-*.md` 覆盖范围内的 Greenfield 场景：

| 场景 | 万亿市场 | 核心挑战 |
|---|---|---|
| Fintech:Payments | 数字银行/支付 | 持牌、资金、AML/KYC、对账、PCI |
| Ecommerce:Marketplace | 电商交易平台 | 平台责任、二清、售后、假货、数据 |
| Autonomous:Fleet | 自动驾驶/智慧城市 | 公共安全、功能安全、法规、责任 |
| SupplyChain:Logistics | 跨境物流 | 海关、Incoterms、制裁、多承运商 |

模拟方法：从一句话需求开始，依次推演 Intake → Clarify → Specify → Gate，重点观察：
1. v5.3 的通用机制能否在**无内置领域知识**时 still 收敛；
2. Agent 会过度推断哪些内容；
3. Gate 会漏掉哪些关键风险。

### 4.2 总体发现

四个 Greenfield 场景的共同模式：

1. **路由判断基本合理**：都能正确落到 `unified_prd + high_risk`（或 `safety_critical` 候选）。
2. **Clarify 阶段严重依赖通用问题**：由于缺乏领域包，Agent 无法主动提出行业特有的 P0 问题，而是发明默认规则（T+1、7 天退货、完全无人化、签收转移责任等）。
3. **Gate 无法识别未声明的推断**：只要 Agent 把推断写成 `RULE-*` 而不标记为 `UNK-*`，gate 就会放行。
4. **产出结构完整但内容不可施工**：模块、角色、状态机、页面 profile 都像样，但核心法规/责任/资金规则可能是错的。

### 4.3 四个场景的产出评分

| 场景 | 评分 | 主要问题 |
|---|---|---|
| Fintech:Payments | 6/10 | 默认监管语境、默认 AML 阈值、缺少 ledger/对账专用 profile |
| Ecommerce:Marketplace | 5/10 | 默认淘宝模式、默认 7 天退货、默认平台 escrow、遗漏二清/假货责任 |
| Autonomous:Fleet | 5/10 | 默认法规/自动化水平、遗漏功能安全/SOTIF/远程接管/地图资质 |
| SupplyChain:Logistics | 5/10 | 默认中国海关/人民币结算、遗漏 Incoterms/制裁筛查/多式联运 |

### 4.4 Greenfield 特有的系统性短板

#### 4.4.1 无领域包时 Clarify 阶段退化

v5.3 §5.4 要求“只追问会改变结果的未知项”，但当可选领域切片不存在时，运行时没有 fallback 领域问题清单。Agent 只能依赖通用模板发问，导致：

- Fintech 遗漏持牌主体、支付状态机、ledger 不可篡改；
- 电商遗漏平台二清、假货连带责任、跨境 VAT；
- 自动驾驶遗漏功能安全/SOTIF、远程接管、地图资质；
- 跨境物流遗漏 Incoterms、制裁筛查、HS 编码责任。

**建议**：为无 domain pack 匹配的高风险/安全关键场景，提供一份 `references/high-risk-patterns.md` 或 `references/greenfield-domain-redlines.md`，覆盖法规、责任、资金、数据保留/跨境、人工接管/降级、保险/赔偿六类 P0 锚点。

#### 4.4.2 Gate 不识别 Agent 的默认推断

v5.3 §10.3 明确 gate 不猜测隐藏未知项。在无领域包时，Agent 会把默认规则写成 `RULE-*` 并通过 gate。例如：

- Fintech 默认 T+1、AML 5 万阈值；
- 电商默认 7 天无理由退货、平台 escrow；
- 自动驾驶默认中国 GB/T 标准、完全无人化；
- 物流默认签收转移责任、人民币结算。

**建议**：
- 当 `assurance_profile=high_risk/safety_critical` 且无匹配领域包时，gate 强制要求所有影响资金、合规、责任的 `RULE-*` 必须引用 `SRC-*` 或标记为 `ASSUMPTION-*`/`UNK-*`；
- 新增 finding code `GAP-DOMAIN-INFERRED-RULE` 或 `RULE-MISSING-SOURCE`，不阻塞但强制人工复核。

#### 4.4.3 页面 profile 缺少行业专用表面

Fintech 需要 `ledger/reconciliation/investigation`，自动驾驶需要 `control_surface/incident_reconstruction/regulatory_export`，物流需要 `customs_declaration/multi_modal_tracking`。当前 7 个 profile 无法覆盖。

**建议**：
- 允许复合 profile（`dashboard+list+workflow`）；
- 新增行业扩展 profile，或在 `prototype.md` 中定义“可扩展 profile”机制，让项目可声明自定义 profile 并配套 gate 规则。

#### 4.4.4 权威模型在 Greenfield 场景下起点过重

v5.3 §6.1 默认配置要求 `SRC-CUSTOMER-CONFIRM-001`、`SRC-CONTRACT-001`、`DEC-AUTHORITY-001`。但 Greenfield 一句话输入下并无这些来源，Agent 可能生成占位 source 或把自身推断包装成客户确认，破坏“Agent 不能自证”原则。

**建议**：当输入无书面来源时，强制创建 `DEC-AUTHORITY-001` 并由用户/责任人显式确认，否则 gate 报 `BLOCKED`。

## 6. 修正建议与版本切割

### 6.1 必须进入 v5.3.0 的修正

以下修正直接影响 v5.3 的核心目标（零 LLM 门禁拦住缺失/漂移/伪完整），应进入 v5.3.0：

1. **扩展页面 profile 机制**
   - 允许复合 profile（`profile=dashboard+list+workflow`）；
   - 新增 `composite/portal`、`builder/designer`、`auth/security` 等基础 profile；
   - 明确 mobile profile 的 `device_capabilities`。

2. **增加横切关注点表示**
   - 增加 `cross-cutting-security.md` / `cross-cutting-ai-runtime.md`；
   - 允许水平 AI 模块 `MOD-AI-xxx`；
   - work packet schema 增加 `ENTITY/REL/DATA-LINEAGE/DICT`。

3. **扩展 gate 检查面**
   - L2+ 增加 `testability` 分支；
   - AI-core + high_risk/safety_critical 增加 AI Runtime Contract 强制检查；
   - 增加 `cross-module-edge`、`data-model`、`unknown-term-scan` 检查器；
   - high_risk 无 domain pack 时强制 `RULE-*` 引用来源。

4. **Coding Agent 交接工程化**
   - 新增 `references/engineering-baseline.md`；
   - 定义 `module-work-packet.schema.json`；
   - 模块 AGENTS.md 前置 TASK-00 检查项。

5. **权威模型 bootstrapping**
   - 提供 `authority_bootstrap` 模式；
   - 支持 `imported_caliber_table` 类型；
   - 法规类来源增加 `jurisdiction/regulatory_version/effective_date/sunset_date`。

### 6.2 建议推迟到 v5.3.1 / v5.4 的项

以下项不影响 v5.3.0 发布可用性，但作为后续增强：

- 完整学习闭环的 promotion validator；
- behavior_validated 实验报告；
- 更多万亿市场 domain candidate；
- Fresh-agent 矩阵扩展；
- 基线 diff 可视化。

### 6.3 建议的版本切割

基于模拟结果，建议调整发布策略：

**v5.3.0（核心收敛版）**：
- 单一 gate 内核 + heading scanner + 四状态 + 中文 finding
- SKILL.md 首屏重写 + references 合并
- 页面 profile 复合机制 + 基础新增 profile
- 横切关注点 packet + 水平 AI 模块模式
- 工程基线阶段 + module work packet schema
- 权威模型 bootstrapping + 法规来源生命周期
- Gate 检查面扩展（testability、cross-module-edge、data-model、AI Runtime）

**v5.3.1（增强版）**：
- 完整 learning loop promotion
- 更多行业 profile 扩展
- Fresh-agent 万亿市场矩阵

这样可以在不降低发布质量的前提下，避免 v5.3.0 承载过多新机制。

## 7. 结论

v5.3.0 的整合方向是正确的：从“叠流程”转向“运行时收敛、门禁归一、证据化学习”。静默双轴路由、统一 PRD、单一确定性 gate、权威模型三元拆分都是扎实的改进。

但经过 16 个角色/场景模拟（12 个已有项目 + 4 个 Greenfield 万亿市场），v5.3 目前的设计成熟度约为 **6/10**。主要短板集中在：

1. **页面 profile 无法覆盖真实复合视图**；
2. **模块 work packet 无法表达横切关注点**；
3. **确定性 gate 检查面太窄**，无法识别 testability、跨模块边、数据模型、AI Runtime、代码级安全和未声明推断；
4. **Coding Agent 交接缺少工程基线**；
5. **权威模型对已有项目和 Greenfield 场景 retrofit 成本过高**；
6. **无内置领域知识时，Agent 会编造默认规则并通过 gate**。

如果按当前设计直接发布 v5.3.0，用户会在实际项目中频繁遇到“gate PASS 但内容错误”“模块切片丢失共享规则”“Coding Agent 按错误假设施工”等问题。建议在进入实现前，先完成 §6.1 中的 5 项核心修正，并相应调整版本切割策略。
