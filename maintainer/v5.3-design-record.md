# AI Delivery Spec v5.3 系列设计与验证记录

> 状态：维护档案，不进入运行时包。完整逐行历史仍由 Git 标签和提交保存。
> 本文件合并原 v5.3 整合设计、多智能体模拟评审、全量修复设计和 v5.3.1 迭代说明，
> 避免维护者目录继续按版本扩散。

## 1. 系列目标

v5.3 不扩张为研发项目管理平台，只强化需求从准入到验收的统一基线：

1. 人类业务、产品、设计、前后端、架构、测试和 Coding Agent 共用一份需求含义；
2. 简单需求保持轻，复杂、高风险和强审计需求按证据自动增强；
3. 存量 PRD/原型先完成 Stage 0，再允许覆盖式重写；
4. 变更、交接与验收沿稳定 ID 双向追溯；
5. 最终门禁保持零 LLM、单遍读取和非生成式；
6. 领域经验只以候选方式回流，不自动晋升为公共规则。

非目标：Sprint/WBS、研发排期、代码生成、CI/CD、部署、监控和运营管理。

## 2. 不变设计决策

### 2.1 静默双轴路由

- `delivery_shape`：需求卡、统一 PRD、受治理事实源；
- `assurance_profile`：有界、标准、高风险、安全关键；
- L0—L4 只作为门禁强度兼容元数据；
- 模块数量不是启用 Product Truth 的充分条件。

### 2.2 一份统一 PRD

- 正文先给30秒摘要、任务阅读地图、角色旅程和模块纵切；
- 每个模块就近闭合页面/数据、规则/权限、状态/事件、指标、恢复、AC 和未知项；
- 附录只索引或精确投影同一批 ID，不另写一套 AI 业务含义；
- 数据上报、审批、批量、集成和高风险需求不能停留在轻量需求卡。

### 2.3 权威与存量盘点

- 来源分为 binding、supporting、contextual、historical 和 untrusted；
- 多个 canonical 候选必须由 `DEC-CONFLICT-*` 裁决；
- Stage 0 将观察项分成 confirmed、inferred、unknown 和 defect_candidate；
- 反推只能形成交互草稿和缺口清单，不能凭原型猜 API、指标、权限或法规。

### 2.4 页面与原型合同

- 页面使用 `primary + layout + surfaces` 组合，而不是行业专用页面模板；
- L3复杂页使用 `REG-*`；所有动作具有 `data-action`、可见/领域结果和 `data-ac`；
- 视觉方向用 feeling、reference、taboo 和 `DEC-AESTHETIC-*`；
- 浏览器执行以 `ARUN-*` 记录，静态门禁不冒充真实交互验收。

### 2.5 Agent 交接控制面

- 根 `AGENTS.md` 只放权威、全局守卫、路由和变更协议；
- `MOD-*` 是可独立负责的模块纵切，并带 `qa_projection`；
- `XCT-*` 负责权限、审计、多租户、血缘、AI运行时等横切约束；
- `EDGE-*` 负责生产者/消费者、映射、重试、幂等和对账；
- 所有工作包绑定同一基线版本/hash、责任人、范围和 AC；
- `ready_for_implementation` 必须引用工程团队维护的工程基线。

### 2.6 大输入与恢复

- 超过输入、字符、模块、页面或 ID 阈值时自动分轮；
- Round 0 固定来源与权威，后续按角色端到端纵切；
- 每轮保存来源 hash、Skill 版本、当前阶段和 ID 切片；
- 中断后只恢复最后有效切片，禁止重新生成巨型事实源。

## 3. 多场景模拟曾暴露的八类问题

| 问题簇 | 最终处置 |
|---|---|
| 复合页面无法表达 | 已采用 `primary + layout + surfaces` |
| 横切规则无工作包 | 已增加 XCT；v5.3.3 补最低完整性门禁 |
| 门禁检查面过窄 | 已扩展页面、数据上报、状态、交接、浏览器证据 |
| Coding Agent 缆少工程基线 | 已用 `engineering_baseline_ref` 处理，不引入研发任务管理 |
| 存量权威初始化成本高 | 已采用来源登记和冲突裁决，不新增 bootstrap 模式 |
| P0 unknown 二元阻断 | 已按阶段和范围返回独立状态/退出码 |
| 路由误分类 | 已加入数据上报、批量、审计、集成等自动升级条件 |
| AI Runtime 易被关键词触发 | 只有真实 AI 行为在范围内才加载对应合同 |

## 5. 质量门禁原则

1. 用户公共 CLI 是唯一需要保证的正式入口，底层验证器测试不能代替 CLI 冒烟测试；
2. 官方示例必须通过对应用户门禁；
3. Finding 必须包含中文原因、修复、最小示例和 RETRY；
4. PASS 只表示声明范围内的确定性合同通过；
5. 浏览器证据、客户签署、领域正确性、代码、安全、性能、部署和生产稳定性分别声明；
6. 测试不得用关键词空壳、假截图路径或固定 PASS 文本证明行为完成。

## 6. 发布与文件预算

- 根目录只保留公共入口和七个功能目录；
- 运行时包通过 `maintainer/runtime-package.yaml` 白名单构建，`maintainer/` 永不进入；
- 运行时预算不超过100个源文件；完整仓库保持低于平台200文件硬限制并预留增长空间；
- 维护设计不再逐版本新增平铺文件，本记录原地更新；完整历史使用 Git 标签查询；
- 测试和证据不能仅为压缩文件数而删除，优先合并重复说明、夹具和输出快照。

## 8. 后续设计约束

- 不新增用户必须理解的模式名；
- 不因为模型能力增强而复制常识性教程；
- 不为了“全生命周期”侵入研发执行与运营工具；
- 不新增中英双份平行文档；入口和高频排障中文优先，稳定标识与通用技术术语保持兼容；
- 新规则优先进入既有 schema/validator/test，只有独立复用价值明确时才新增文件。

## 10. v5.3.3 需求卡生命周期回归（2026-07-22）

从五个已知项目各抽取两个真正有界的小功能，共形成 10 张独立需求卡：安驾融合的资源格式筛选、仅看我创建的课程；神通 CRM 的线索来源筛选、仅看超期工单；数据报告的模板创建人筛选、生成失败筛选；人车企上报的上次成功时间排序、复制失败关联号；SaaS 知识库的仅看我创建、复制资产编号。

- 5 张走“准入→澄清关闭→规格→评审→基线→验收→关闭”的直线路径；
- 5 张额外走“变更申请→影响分析→审批同步→重建基线→验收→关闭”的变更路径；
- 每张卡均由结构化 intake 自动分诊为 `requirement_card/bounded`，L1 规格门禁 PASS；
- 每张卡均生成关闭态 requirement register，并用正向与恢复路径两个验收项验证 ARUN；
- 所有证据明确标注 `LAB SIMULATION ONLY`，仅证明生命周期机制和契约可运行，不代表对应项目功能已经开发或经客户验收；
- 测试数据在临时目录生成，不增加运行时文件和仓库样例负担。
