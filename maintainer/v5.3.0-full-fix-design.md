## 1. 继承 v5.3 原设计的正确方向（保持不变）

以下来自 `v5.3-integrated-detailed-design.md` 的核心架构，不修改：

| 机制 | 实现 | 理由 |
|------|------|------|
| 静默双轴路由 | `triage_requirement.py` 内部分诊，不暴露 mode×tier 矩阵给 Agent | 降低认知负担 |
| 统一 PRD | 单一 `unified_prd.md` 含人读 + 机读层 | 减少文件碎片 |
| 单一 gate 内核 | `quality_gate.py` 一条 `run(profile, path)` 入口，四个状态 | 防 Agent 自证合格 |
| 四状态终态 | PASS / BLOCKED / BLOCKED_BY_P0_UNKNOWN / NEEDS_CLARIFY | 语义明确 |
| 权威模型三元拆分 | 写入面 / 约束来源 / 基线审批 替代旧 `authority_mode` | 更精确 |
| 双通道分发 | 运行包（精简到 ~35 文件）与完整仓库并存 | 解决 npm/文件数限制 |
| 中文 finding | finding.message 和 fix 提示均为中文 | 降低中文团队门槛 |
| 段级领域检索 | `## 领域名` 标题匹配精确加载 | 大领域包不必整文件吞 |

### P0-001：无领域包时 high_risk RULE 必须溯源

**问题**：Agent 在无 domain pack 匹配时将自身推断写成 `RULE-*` 并通过 gate（如 Fintech 默认 T+1）。

**修复方案**（gate 条件分支，~40 行）：

```python
# quality_gate.py 新增
def _check_rule_sourcing(unified_prd, gate_profile):
    """当 assurance_profile 为 high_risk/safety_critical 且无匹配领域包时，
    所有影响资金、合规、责任的 RULE-* 必须引用 SRC-* 或标记为 ASSUMPTION-*"""
    if gate_profile.get("assurance_profile") not in ("high_risk", "safety_critical"):
        return []
    if gate_profile.get("domain_pack_matched"):
        return []  # 有领域包时信任领域包的默认规则
    
    findings = []
    critical_rule_patterns = [
        "资金", "支付", "结算", "对账", "退款",
        "合规", "监管", "牌照", "持牌", "AML", "KYC",
        "责任", "赔偿", "保险", "担保",
        "数据", "隐私", "跨境", "GDPR", "PIPL"
    ]
    for rule_id, rule in _extract_rules(unified_prd):
        if not any(kw in str(rule).lower() for kw in critical_rule_patterns):
            continue
        if not _has_source_ref(rule):
            findings.append(Finding(
                code="RULE-MISSING-SOURCE",
                level="BLOCKER",
                entity=rule_id,
                message=f"高风险场景无领域包：规则 {rule_id} 涉及资金/合规/责任，须引用 SRC-* 或标记 ASSUMPTION-*",
                fix="为该规则添加 binding_sources 引用，或将 Agent 推断标记为 ASSUMPTION-{id}",
                retry="quality_gate --profile prd --level L2 --fix RULE-MISSING-SOURCE"
            ))
    return findings
```

**文件影响**：`quality_gate.py` +40 行，无新文件。

### P0-003：代码安全基检（参数化查询 / 输入校验）

**问题**：Gate 不检查代码层安全，SQL 注入等缺陷会进入实现。

**修复方案**（确定性文本模式扫描，~50 行）：

```python
def _check_code_safety(unified_prd):
    """确定性黑名单扫描：字符串拼接 SQL、裸用户输入"""
    findings = []
    patterns = [
        (r'(?i)(execute|exec|query)\s*\(\s*["\'].*\+', "CODE-SQL-CONCAT",
         "疑似 SQL 字符串拼接，建议使用参数化查询"),
        (r'(?i)(request|input|params)\s*\[.*\]\s*(?!.*escape|.*saniti)', "CODE-RAW-INPUT",
         "用户输入未转义，存在注入风险"),
        (r'(?i)eval\s*\(\s*(?!.*json)', "CODE-DYNAMIC-EVAL",
         "动态 eval 调用，可能执行不可信代码"),
    ]
    for code_block_id, content in _extract_code_blocks(unified_prd):
        for pattern, code, desc in patterns:
            if re.search(pattern, str(content)):
                findings.append(Finding(
                    code=code, level="BLOCKER", entity=code_block_id,
                    message=desc,
                    fix=f"将 {code_block_id} 中的动态拼接替换为参数化方式",
                    retry="quality_gate --profile prd --fix CODE-SAFETY"
                ))
    return findings
```

**文件影响**：`quality_gate.py` +50 行，无新文件。

### P0-007：隐性未知项扫描

**问题**：标注为"已确认"的区域残留 TBD/stub/待实现，gate 不识别。

**修复方案**（确定性文本匹配，~25 行）：

```python
UNKNOWN_TERM_PATTERNS = [
    r'(?i)\bTBD\b', r'(?i)\bTODO\b', r'(?i)\bFIXME\b',
    r'(?i)\bstub\b', r'(?i)\bplaceholder\b',
    r'待确认', r'待实现', r'待补充', r'暂未定义', r'留空'
]

def _scan_unknown_terms(unified_prd):
    """扫描 PRD 全文中的隐性未知项标记"""
    for pattern in UNKNOWN_TERM_PATTERNS:
        for match in re.finditer(pattern, unified_prd):
            line = unified_prd[:match.start()].count('\n') + 1
            yield Finding(
                code="UNKNOWN-TERM-FOUND",
                level="BLOCKER",
                entity=f"L{line}",
                message=f"第 {line} 行发现隐性未知项标记：{match.group()}",
                fix="将其声明为 UNK-xxx 并标记 P0/P1 优先级，或替换为确认内容后删除标记",
                retry="quality_gate --profile prd --fix UNKNOWN-TERM"
            )
```

**文件影响**：`quality_gate.py` +25 行。
